# 0006 — Логические правила маршрутизации (AND/OR/NOT) с рекурсивным деревом под-условий

- **Статус:** Accepted
- **Дата:** 2026-06-16
- **PR:** #114

## Context

Модель маршрутного правила была плоским одиночным матчером:
`RoutingRule{Type, Value, ProviderID, NoResolve, Target}` — один тип, одно значение, один
таргет. Она не выражает логические/составные правила mihomo (`AND`/`OR`/`NOT`), у которых
payload — это вложенный список под-условий: `LOGIC,((TYPE,VAL),(TYPE,VAL)),TARGET`. Это
блокировало реальную операционную потребность — глушение QUIC

```
AND,((NETWORK,UDP),(DST-PORT,443)),REJECT-DROP
```

(заставляет HTTP/3 откатиться на TCP; чинит подвисания App Store / YouTube под TUN+fake-ip)
— его нельзя было собрать из UI.

Таргеты уже типизированы через `PolicyRef` (см. [ADR-0005](0005-strict-mihomo-refs.md)) —
пробел был исключительно на стороне матчеров. Дополнительно: оператор мог вписать секцию
`sub-rules` в base YAML и словить коллизию с генерацией (ключ не был в `GeneratedKeys`).

Сверка реестра `ruleTypes` с вики mihomo показала ещё четыре отсутствующих простых
матчера: `SRC-IP-ASN`, `SRC-IP-SUFFIX`, `PROCESS-PATH-WILDCARD`, `PROCESS-NAME-WILDCARD`.

Ограничения, выясненные из исходника mihomo (`rules/logic`): `NOT` принимает ровно одно
под-условие; под-условия парсятся через `ParseRulePayload(payload, parseParams=false)` —
**параметры (в т.ч. `no-resolve`) внутри логического под-условия не извлекаются**, то есть
под-условие не несёт `no-resolve`. `SUB-RULE` (ссылка на именованную группу sub-rule) —
отдельная крупная фича (именованные группы под-правил = второй редактор); по решению
владельца в этой итерации **не реализуется**, только закрывается дыра base YAML.

## Considered Options

### Хранение рекурсивного дерева под-условий

- **A. Типизированная реляционная таблица** `mihomo_rule_conditions` (parent-self-ref,
  `provider_id` — настоящий FK). Плюсы: согласовано с остальной схемой (`provider_id`
  внутри условия — такой же FK, как у плоского RULE-SET), инвариант «typed value, не
  строка» сохранён; CASCADE по `rule_id`. Минусы: рекурсия в трёх местах (save с резолвом
  id, read со сборкой дерева, clone с remap rule/parent/provider).
- **B. JSON-блоб** под-условий в колонке правила. Плюсы: один столбец, без рекурсивных
  SQL. Минусы: ссылки на провайдер внутри блоба — не FK (индекс/имя строкой), что прямо
  нарушает правила проекта (entity по id, никаких «магических строк»); «не free-text-блоб»
  — явное требование задачи.

### Форма под-условия в OpenAPI

- **A. Отдельный тип `MihomoCondition`** (рекурсивный, без `target`/`noResolve`). Честно
  отражает, что под-условие — матчер, а не правило.
- **B. Переиспользовать `MihomoRule`** с необязательным `target` и проверкой «target
  обязателен на верхнем уровне» в коде. Меньше типов, но ослабляет контракт правила и
  тащит неприменимые поля в под-условие.

### `no-resolve` у под-условий

- **A. Не хранить и не рендерить** (mihomo всё равно не парсит параметры под-условий).
- **B. Хранить по типу** (как у плоского правила) — но mihomo это проигнорирует, а в YAML
  попадёт мусор.

## Decision

- **Хранение — реляционная таблица (A).** `mihomo_rule_conditions(id, rule_id→rules
  CASCADE, parent_id→self CASCADE, position, type, value, provider_id→providers)`. Логическое
  правило верхнего уровня остаётся строкой `mihomo_routing_rules` (`type` = AND/OR/NOT,
  `value`/`provider_id` NULL), его дети — строки с `parent_id IS NULL`; вложенные — с
  `parent_id` на владеющее условие. JSON-блоб отвергнут как нарушающий инвариант
  типизированных ссылок. Это и есть «реальная работа по БД», которую задача требовала.
- **OpenAPI — отдельный `MihomoCondition` (A).** Рекурсивный self-`$ref`, поля
  `type`/`value`/`providerIdx`/`conditions` — без `target`/`noResolve`. `MihomoRule` получает
  `conditions[]`. Так контракт правила не ослаблен.
- **`no-resolve` у под-условий — не существует (A).** Под-условие = `{type, value,
  provider, children}`. Это и корректнее (совпадает с парсером mihomo), и минималистичнее
  (нет столбца/поля).
- **Доменные типы:** `RuleCondition{Type, Value, ProviderID, Conditions}` (само-рекурсивный),
  `RoutingRule.Conditions []RuleCondition`; на save — `ConditionDraft` (провайдер по индексу,
  как остальной draft-граф). Логический признак — флаг `RuleTypeOptions.Logical` +
  `RuleType.IsLogical()`, никаких сравнений подстрок.
- **Валидация (рекурсивная, в коде — не в схеме):** `NOT` ровно одно условие
  (`ErrNotArity`), `AND`/`OR` минимум два (`ErrLogicalArity`), `MATCH` нельзя как условие
  (`ErrConditionMatch`), у простого правила нет условий (`ErrConditionsNotAllowed`); у
  логического нет value/provider/no-resolve; provider-индекс под-условия в диапазоне.
- **`sub-rules` → `GeneratedKeys`.** Закрывает дыру base YAML (save отвергает, render
  стрипает); subgen именованные sub-rule-группы не генерирует. SUB-RULE как тип правила не
  добавлен.
- **Паритет матчеров:** + `SRC-IP-ASN`, `SRC-IP-SUFFIX`, `PROCESS-PATH-WILDCARD`,
  `PROCESS-NAME-WILDCARD` (простые, без особых опций).
- **UI:** рекурсивный компонент `condition-node` (дерево с отступами как у group-members).
  Переупорядочивание под-условий **намеренно не реализовано** — AND/OR/NOT коммутативны,
  порядок не влияет на матчинг (и это снимает конфликт вложенных SortableJS с одинаковыми
  селекторами). Перетаскивание правил верхнего уровня сохранено (строка+условия обёрнуты в
  единый перетаскиваемый `.rule-item`).

## Consequences

- Произвольная рекурсия `AND(AND(OR(...),NOT(...)),...)` round-trip'ит БД ↔ render ↔
  decode; QUIC-правило собирается из UI и рендерится дословно.
- `mihomo_rule_conditions` добавлена аддитивной миграцией `0004-*.sql` (CREATE в
  транзакции, без rebuild). Save читает id правил обратно и вставляет дерево по-узлово
  (`LastInsertId` для `parent_id`); read собирает дерево в памяти; clone ремапит
  rule/parent/provider. Конфиги крошечные — по-узловые вставки несущественны.
- Под-условия не ссылаются на инбаунды (это матчеры, не таргеты) → per-subscriber drop
  работает только по `Target` правила; единственный «сбой» рендера под-условия — нерезолвнутый
  провайдер RULE-SET (config-ошибка, лог + drop правила).
- Под-условия не несут `no-resolve` — если он понадобится логике mihomo в будущем, это
  потребует и поддержки в парсере mihomo, и расширения модели.
- Порядок под-условий в UI не редактируется; если когда-нибудь понадобится — добавляется
  тривиально (модель уже упорядочена `position`).
- SUB-RULE (именованные группы под-правил) остаётся будущей задачей; `sub-rules`
  зарезервирован за subgen, так что ввести его позже можно без миграции ключа.
